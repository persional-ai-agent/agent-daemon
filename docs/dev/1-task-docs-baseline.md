## 1. 目标

- 要解决的问题：仓库缺少可复用的产品/开发总览文档，`docs/dev` 索引未初始化，导致后续需求缺少稳定上下文入口。
- 成功标准：
  - 基于现有代码与子项目文档，沉淀当前已实现功能边界与核心流程；
  - 形成 `docs/overview-product.md` 与 `docs/overview-product-dev.md`；
  - 初始化 `docs/dev/README.md` 索引并登记本任务。

## 2. 背景与边界

- 背景：仓库已有较完整实现（daemon、cli、api、gateway、plugin、cron、web、tui），但文档目录仅有空 `docs/dev`。
- 包含：
  - 只读分析代码与现有 README；
  - 新增/更新文档文件（不改业务代码）。
- 不包含：
  - 功能实现、接口改造、测试逻辑变更。
- 关键假设：代码中的路由、命令、结构体和 README 能代表当前“已实现”能力。
- 非目标：为所有模块写逐函数说明或补全教程式文档。
- 最大修改范围：`docs/` 下新增 4 个文档文件。
- 禁止触碰范围：`cmd/`、`internal/`、`web/src/`、`ui-tui/*.go` 业务代码。

## 3. 风险门禁

| 项 | 结论 |
|----|------|
| 风险矩阵 | L2（单模块文档新增，但跨产品/开发事实沉淀，需可追溯记录） |
| 高风险开发门禁 | 否 |
| 破坏性操作 | 否 |
| 用户已有修改 | 否（`git status --short` 为空） |
| 底层/系统风险 | 否 |
| 命令权限 | C0/C1（只读分析 + 文档写入） |
| 用户确认事项 | 无 |
| 止损/回滚方式 | 删除新增文档或按文件级别 git 回退 |

## 4. 方案

- 推荐方案：按模板建立“总览 + 索引 + 单任务记录”最小文档体系，所有事实均来自当前代码/README，可直接被后续任务复用。
- 取舍理由：相比一次性写大量专题文档，此方案上下文成本更低、维护入口更清晰。
- 风险与应对：
  - 风险：文档内容与未来代码演进偏离。
  - 应对：在变更记录中显式标记来源任务，后续按任务持续更新总文档。

## 5. 执行计划

1. 盘点代码与现有 README 的功能边界 -> 验证：命令/路由/模块可在源码定位。
2. 编写产品总览文档 -> 验证：覆盖定位、边界、场景、流程、规则与索引。
3. 编写开发总览文档 -> 验证：覆盖技术栈、架构、接口、数据、验证与发布回滚。
4. 初始化 `docs/dev/README.md` 并登记任务 -> 验证：编号规则与索引记录完整。
